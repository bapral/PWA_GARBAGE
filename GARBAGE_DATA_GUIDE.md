# 台灣五都垃圾車整合開發指南 (2026 修訂版)

本文件旨在協助開發者快速理解本專案如何介接、處理並展示台灣主要城市（台北、新北、台中、台南、高雄）的垃圾清運資料。

---

## 1. 資料處理哲學：三層架構

本專案將資料處理分為三個層級，以平衡「即時性」與「穩定性」：

1.  **API 即時層 (Realtime Layer)**: 
    *   **來源**: 政府開放資料即時 API。
    *   **頻率**: 每 30 秒輪詢一次。
    *   **挑戰**: 跨網域限制 (Web)、伺服器不穩定、資料格式雜亂。
2.  **資料庫推估層 (Prediction Layer)**:
    *   **來源**: 本地 SQLite 資料庫。
    *   **觸發**: 當 API 無回應、網路斷線或非即時運作時段。
    *   **邏輯**: 根據目前時間，檢索資料庫中「理論上」應在該點位出現的車輛。
3.  **內建資產層 (Assets Layer)**:
    *   **來源**: `assets/*.json` 或 `assets/*.csv`。
    *   **觸發**: 首次安裝或資料庫損毀時。
    *   **目的**: 確保應用程式在完全無網路環境下仍能顯示基本的清運點位。

---

## 2. 各縣市特異性實作細節

### 台北市：巨量資料與分區策略
*   **分頁抓取**: 台北市班表超過 4000 筆，單次請求 limit 預設不足，需實作 `offset` 分段同步。
*   **複合 ID**: 路線名可能重複，必須組合 `路線 + 車號` 作為 Model Key。

### 新北市：安全門檻與 Isolate
*   **User-Agent**: 嚴格檢查標頭，未設定模擬瀏覽器會回傳 403。
*   **背景解析**: 新北 CSV 體積巨大（>10MB），嚴禁在 UI Thread 解析，否則地圖會凍結。

### 台中市：車牌映射與 ISO T
*   **Linkage**: 使用「車牌號碼」作為動態位置與靜態班表的唯一關聯。
*   **時間解析**: 需處理 `20240411T093000` 字串。

### 南台灣 (台南、高雄)：格式混雜與備援
*   **座標拆分**: 高雄 API 座標常以 `"22.6,120.3"` 形式存在，需手動 `split`。
*   **大小寫不一**: 處理 `LATITUDE` (台南靜態) 與 `y` (台南即時) 的映射。

---

## 3. Web 平台 (PWA) 的特殊考量

由於政府 API 多未開啟 CORS 支援，本專案在 Web 端採取以下策略：

1.  **多代理輪詢 (Proxy Polling)**: 內建 `CorsProxy.io`、`AllOrigins`、`CodeTabs` 等多個代理伺服器。
2.  **自動 JSON 拆包**: 自動偵測代理伺服器回傳的內容（有些會封裝在 `contents` 欄位中）。
3.  **WASM SQLite**: 透過 `sqflite_common_ffi_web` 實現瀏覽器端的持久化資料庫存取。

---

## 4. 維護建議

*   **更新班表**: 若發現某城市路線大改，應更新 `assets/` 下的檔案，並提高對應 Service 內的 `requiredAssetVersion`。
*   **座標偏離**: 若發現位置偏移數百公尺，請確認 API 是否切換為 TWD97 座標系（本專案目前採用 WGS84）。
